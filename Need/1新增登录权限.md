## 管理员登录安全加固
为了给管理员登录加上真正的“铁布衫”，在企业级后台架构中，我们通常会叠加以下 两层防御机制：


### 第一层：账号锁定机制（防暴力破解底线）
光靠图形验证码拦截是不够的，必须在 Redis 里做真正的密码错误次数熔断。

逻辑：

管理员连续输错 5 次密码。

账号被直接锁定 30 分钟（或更久），期间即使密码输入正确也拒绝登录。

Redis 实现规范：

Key：haifeng:admin:login:fail:{phone}

Value：错误次数（Integer）

TTL（过期时间）：30分钟。每次输错 INCR 递增，达到 5 次抛出异常。登录成功则 DEL 清除该 Key。


### 第二层：MFA 双因素认证（动态令牌 TOTP）
这是目前后台管理系统最主流、最安全且零成本的终极防御方案。

什么是 TOTP？ 就是像“谷歌身份验证器 (Google Authenticator) ”或“微软身份验证器”那种每 30 秒变一次的 6 位数字动态密码。

数据库修改需求：
要在你的 sys_admin 表里增加两个字段来支持它：

SQL
ALTER TABLE sys_admin ADD COLUMN totp_secret VARCHAR(64);
ALTER TABLE sys_admin ADD COLUMN is_totp_enabled BOOLEAN DEFAULT FALSE;

COMMENT ON COLUMN sys_admin.totp_secret IS 'TOTP动态口令密钥(用于绑定Google Authenticator)';
COMMENT ON COLUMN sys_admin.is_totp_enabled IS '是否已开启双因素认证';
登录流程变化：

前端先传 phone + password + captcha。

后端校验通过后，不发 Token，而是返回一个状态码（如 code: 20001, msg: "需进行二次验证"）。

前端弹出一个输入框，管理员掏出手机，输入 6 位动态码提交给后端。

后端校验动态码正确，才真正发放 JWT Token。

### 流程
阶段0：管理员首次开启 TOTP（仅一次）
管理员登录后台，进入“安全设置”页面，点击「开启动态验证码」。

后端生成一个固定的密钥（totp_secret），保存到该管理员的数据库记录中。

后端用这个密钥生成一个二维码（内容包含密钥、用户名、你的系统名称）。

管理员用手机上任意 TOTP 类 App 扫描二维码。

管理员在网页上输入 App 里显示的6位数字。

后端验证通过后，标记 is_totp_enabled = true。

自此以后，管理员手机里的 App 会一直基于那个密钥每秒变化数字。

阶段1：日常登录（两步提交）
管理员输入手机号+密码 → 后端验证通过 → 发现 is_totp_enabled = true → 不直接发 JWT，而是生成一个临时凭证（pre-auth token）存在 Redis，返回 20001 和这个临时凭证。

前端弹出输入框，要求管理员输入手机上实时看到的6位数字。

管理员输入后，前端调用 /login/totp 接口，带上临时凭证 + 6位码。

后端用数据库里存的那个 同一个密钥 去验证6位码是否正确。

### 注意事项
1. 为了防止管理员手机上totp丢失，我们让管理员拥有权限，他可以检查管理员的列表，新增修改管理员的启用状态。
2. 需要向前端返回totp_secret，让前端生成二维码。二维码不是在每次登录时生成，而是在管理员第一次开启 TOTP 时生成一次。
3. 后台管理员可以查看自己的二维码，嘶，这就需要管理员的个人中心了，展示自己的管理员表信息对吧，只允许查，修改只能高级管理员可以改
