# 约束匹配模块接口文档

> 基础路径：`/api/v1/app/constraint`
> 
> 所有接口需要登录（`@RequireLogin`）
> 
> 所有接口有频率限制（基于 IP），超频返回 `429`

---

## 1. 获取当前用户触发的约束列表

根据当前用户的高考档案（身体条件、身份条件、选科等），匹配所有激活的约束规则，返回用户"触发"了哪些约束。

**触发含义：** 用户的档案数据满足了某条约束的判定条件（如色盲、色弱、嗅觉迟钝等），则该约束被触发。

### 请求

```
GET /api/v1/app/constraint/match
```

### 请求参数

无

### 频率限制

- 60 秒内最多 30 次

### 响应示例

**有触发约束：**
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "constraintCodes": ["COLOR_BLIND", "COLOR_WEAK", "SMELL_DISORDER"]
  }
}
```

**无触发约束：**
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "constraintCodes": []
  }
}
```

### 响应字段说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| constraintCodes | List\<String\> | 触发的约束代码列表 |

### 错误码

| code | 说明 |
|------|------|
| 200 | 成功 |
| 401 | 未登录 |
| 1010 | 用户高考档案不存在（需先填写档案） |
| 429 | 请求过于频繁 |

## 2. 根据约束代码列表获取约束详情

批量查询约束字典的详细信息。用于前端展示约束的具体描述、分类等。

### 请求

```
POST /api/v1/app/constraint/details
Content-Type: application/json
```

### 请求体

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| codes | List\<String\> | 是 | 约束代码列表，最多 100 个，每个代码最多 50 字符 |

### 请求体示例

```json
{
  "codes": ["COLOR_BLIND", "COLOR_WEAK", "SMELL_DISORDER"]
}
```

### 频率限制

- 60 秒内最多 20 次

### 响应示例

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "constraints": [
      {
        "code": "COLOR_BLIND",
        "name": "色盲",
        "category": "身体视觉条件",
        "description": "色盲考生不能录取到以颜色作为严格标准的专业，如美术、医学、化学等",
        "severity": "HARD"
      },
      {
        "code": "COLOR_WEAK",
        "name": "色弱",
        "category": "身体视觉条件",
        "description": "色弱考生不宜录取到以颜色作为严格标准的专业",
        "severity": "HARD"
      }
    ]
  }
}
```

**空列表：**
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "constraints": []
  }
}
```

### 响应字段说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| constraints | List | 约束详情列表 |
| constraints[].code | String | 约束代码（唯一标识） |
| constraints[].name | String | 约束名称 |
| constraints[].category | String | 约束大类（如"身体视觉条件"、"身份条件"等） |
| constraints[].description | String | 约束描述/提示信息 |
| constraints[].severity | String | 严重程度：`HARD`（硬限制，不可报考）或 `SOFT`（软提示，建议谨慎） |

> **注意：** 传入的 codes 中如果包含不存在或已删除的代码，不会报错，只是结果中不包含该代码对应的详情。

## 3. 校验当前用户是否满足专业组约束

检查当前用户触发的约束与指定专业组的约束要求是否有冲突。用于判断用户是否可以报考该专业组。

### 请求

```
POST /api/v1/app/constraint/check-group
Content-Type: application/json
```

### 请求体

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| groupId | Integer | 是 | 专业组 ID，必须大于 0 |

### 请求体示例

```json
{
  "groupId": 123
}
```

### 频率限制

- 60 秒内最多 20 次

### 响应示例

**通过（无冲突）：**
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "isPass": true,
    "hardConflicts": [],
    "softConflicts": []
  }
}
```

**不通过（有硬限制冲突）：**
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "isPass": false,
    "hardConflicts": [
      {
        "code": "COLOR_BLIND",
        "name": "色盲",
        "description": "色盲考生不能录取到以颜色作为严格标准的专业"
      }
    ],
    "softConflicts": []
  }
}
```

**仅有软提示（可报考但建议谨慎）：**
```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "isPass": true,
    "hardConflicts": [],
    "softConflicts": [
      {
        "code": "COLOR_WEAK",
        "name": "色弱",
        "description": "色弱考生不宜录取到以颜色作为严格标准的专业"
      }
    ]
  }
}
```

### 响应字段说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| isPass | Boolean | 是否通过校验。`true` = 无 HARD 冲突，可以报考；`false` = 有 HARD 冲突，不可报考 |
| hardConflicts | List | 硬限制冲突列表（不可报考） |
| softConflicts | List | 软提示冲突列表（建议谨慎） |
| hardConflicts[].code | String | 约束代码 |
| hardConflicts[].name | String | 约束名称 |
| hardConflicts[].description | String | 约束描述 |

### 错误码

| code | 说明 |
|------|------|
| 200 | 成功 |
| 400 | 参数错误（groupId 为空或非正数） |
| 401 | 未登录 |
| 1010 | 用户高考档案不存在 |
| 1011 | 专业组不存在 |
| 429 | 请求过于频繁 |

## 注意事项

### 1. 前置条件

调用约束相关接口前，用户**必须先填写高考档案**（`POST /api/v1/app/gaokao/archive`）。未填写档案会返回错误码 `1010`。

### 2. 约束匹配逻辑

- 系统从 `t_constraint_dict` 表加载所有 `is_active = true` 且 `is_deleted = false` 的约束规则
- 每条规则定义了判定条件（`check_field` + `check_operator` + `check_value`）和可选的额外条件（`extra_field` + `extra_operator` + `extra_value`）
- 主条件与额外条件为 **AND** 关系（同时满足才触发）
- 匹配结果会缓存 5 分钟，超时后自动重新加载

### 3. 冲突判定逻辑

`check-group` 接口的冲突判定：
- 用户触发的约束 = 用户具有某种限制性特征
- 专业组约束 = 该专业组的限制性要求
- **冲突 = 用户有某个限制特征 AND 该专业组也有这个限制** → 用户不符合该专业组
- `isPass = true` 当且仅当 `hardConflicts` 为空

### 4. severity 说明

| 值 | 含义 | 对前端的影响 |
|----|------|------------|
| `HARD` | 硬限制 | 冲突时 `isPass = false`，前端应阻止用户选择该专业组 |
| `SOFT` | 软提示 | 冲突时 `isPass` 仍为 `true`，但前端应展示提示信息让用户知晓风险 |

### 5. 频率限制

所有接口基于 IP 做频率限制，使用 Redis 计数器实现：

| 接口 | 限制 |
|------|------|
| `GET /match` | 30 次/60 秒 |
| `POST /details` | 20 次/60 秒 |
| `POST /check-group` | 20 次/60 秒 |

超频返回 HTTP 429，响应体：
```json
{
  "code": 429,
  "msg": "请求过于频繁",
  "data": null
}
```

### 6. 推荐调用流程

```
1. 用户填写高考档案（POST /api/v1/app/gaokao/archive）
2. 调用 GET /constraint/match 获取用户触发的约束列表
3. 前端展示用户有哪些限制性特征（如"色盲"、"色弱"）
4. 用户浏览专业组列表时，调用 POST /constraint/check-group 判断每个专业组是否可报
5. 用户点击某个约束代码时，调用 POST /constraint/details 获取详细描述
```

### 7. 缓存说明

- 约束字典数据缓存在 Redis 中，Key 为 `haifeng:constraint:active:list`，TTL 5 分钟
- 管理员在后台增删改约束字典时，需调用缓存清除接口使缓存失效；否则需等待 5 分钟自然过期
- 缓存读写异常时自动降级查数据库，不影响业务
