# 短信验证码 - 忘记密码功能 · 需求文档

> 版本：v1.0  
> 日期：2026-07-06  
> 状态：待评审

---

## 1. 需求概述

### 1.1 背景
目前项目中用户（C端会员）只有「已登录修改密码」功能，缺少「忘记密码」的找回途径。  
管理员（B端）使用 TOTP 双因素认证，不需要短信。**本功能仅面向 C端会员用户。**

### 1.2 目标
- 用户通过手机号接收短信验证码，验证身份后重置密码
- 防止黑客恶意刷短信、暴力破解验证码

---

## 2. 业务流程

```
Step 1: 用户输入手机号 → 请求发送验证码
               ↓
Step 2: 系统校验（手机号存在？60s冷却？日限？）
               ↓
Step 3: 调用 Submail API 发送6位数字验证码
               ↓
Step 4: 用户输入验证码 + 新密码 → 提交重置
               ↓
Step 5: 系统校验验证码 → 更新密码 → 返回成功
```

### 2.1 接口列表

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/app/auth/forgot-password/send-code` | 发送短信验证码 |
| POST | `/api/v1/app/auth/forgot-password/reset` | 校验验证码并重置密码 |

**安全白名单：** 以上两个接口加入 `SecurityConfig` 白名单（免登录访问）。

## 3. 接口详细设计

### 3.1 发送验证码

**Request:**

```json
{
  "phone": "13800138000",
  "captchaCode": "a3b2",
  "uuid": "abc-123-def"
}
```

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| phone | String | 是 | `^1[3-9]\d{9}$` |
| captchaCode | String | 是 | 4位（复用现有图形验证码规则） |
| uuid | String | 是 | 图形验证码的UUID |

**校验逻辑（顺序执行，任一失败即返回错误）：**

1. **图形验证码校验** — 防止机器刷接口，复用现有 `CaptchaService` 校验 `captchaCode` + `uuid`
   - 一次性使用，校验后删除 Redis key
   - 失败 → 返回 `400`
2. **手机号是否存在** — 查询 `t_member` 表。不存在 → 统一返回 `200`，**不提示"手机号未注册"以避免枚举攻击**
   - 具体处理：手机号不存在时，假装发送成功，但不实际调用 Submail、不存储验证码
3. **60秒发送冷却** — Redis key: `haifeng:sms:send:cool:{phone}`，TTL=60s
   - 如果 key 存在 → 返回 `429`（请求过于频繁）
4. **每日发送次数限制** — Redis key: `haifeng:sms:send:limit:{date}:{phone}`，计数器
   - 超过5次 → 返回 `429`
5. **IP 频率限制** — Redis key: `haifeng:limit:api:{ip}:/api/v1/app/auth/forgot-password/send-code`
   - 使用现有 `haifeng:limit:api` 模式（已在 RedisKeyConstant 中定义）

**通过所有校验后：**
- 生成 6 位随机数字验证码（如 `438921`）
- 调用 Submail API `sms/send` 发送短信
- 验证码存入 Redis：`haifeng:sms:code:{phone}`，value=验证码，TTL=5分钟
- 设置60秒冷却 key：`haifeng:sms:send:cool:{phone}`，TTL=60s
- 递增日限计数器：`haifeng:sms:send:limit:{yyyyMMdd}:{phone}`

**Response:**

```json
{
  "code": 200,
  "msg": "success",
  "data": null,
  "timestamp": 1234567890
}
```

> 无论手机号是否存在，统一返回 200，避免攻击者通过响应差异枚举已注册手机号。

### 3.2 重置密码

**Request:**

```json
{
  "phone": "13800138000",
  "code": "438921",
  "password": "newPwd123"
}
```

| 字段 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| phone | String | 是 | `^1[3-9]\d{9}$` |
| code | String | 是 | 6位数字 |
| password | String | 是 | 字母+数字，6-16位（复用现有 `RegisterDTO` 的密码正则） |

**校验逻辑（顺序执行）：**

1. **手机号是否存在** — 查询 `t_member`。不存在 → 返回错误码 `1001`
2. **验证码是否存在** — Redis key `haifeng:sms:code:{phone}`。不存在或已过期 → `400` "验证码已过期"
3. **验证码匹配** — 比对用户输入的 code 与 Redis 中的值
   - 匹配成功 → 删除 Redis 中的验证码（一次性使用）
   - 匹配失败 → 递增失败计数器 `haifeng:sms:verify:fail:{phone}`。连续5次失败 → 锁定30分钟（key TTL=1800s），锁定期间返回 `429`
4. **密码强度校验** — 使用 `@Pattern` 注解（字母+数字，6-16位）
5. **更新密码** — BCrypt 加密后更新 `t_member.password`
6. **清除该用户所有登录态** — 删除 Redis 中 `haifeng:token:refresh:member:{userId}`，强制用户重新登录

**Response:**

```json
{
  "code": 200,
  "msg": "success",
  "data": null,
  "timestamp": 1234567890
}
```

## 4. 安全策略汇总

| 防护维度 | 规则 | 参考 Redis Key |
|---------|------|----------------|
| 发送冷却 | 同一手机号间隔 ≥ 60秒 | `haifeng:sms:send:cool:{phone}` |
| 每日上限 | 同一手机号每天最多5次 | `haifeng:sms:send:limit:{date}:{phone}` |
| IP限流 | 同一IP对 send-code 接口限流 | `haifeng:limit:api:{ip}:{path}` |
| 验证码暴力破解 | 连续5次错误锁定30分钟 | `haifeng:sms:verify:fail:{phone}` |
| 手机号枚举防御 | 发送接口统一返回200 | — |
| 一次性验证码 | 验证成功后立即删除 Redis key | `haifeng:sms:code:{phone}` |
| 图形验证码前置 | send-code 需要先通过图形验证码 | 复用现有 CaptchaService |

## 5. 短信模板规格

### 5.1 Submail 短信发送

使用 Submail `sms/send` 接口（非模板发送），POST 到：
```
https://api-v4.mysubmail.com/sms/send.json
```

请求参数（以 `normal` 模式为例，直接用 appkey 作 signature）：

| 参数 | 值 |
|------|-----|
| appid | `${submail.appid}`（环境变量配置） |
| to | 用户手机号 |
| content | `【海峰规划】您的验证码是：{code}，5分钟内有效，请勿泄露他人。` |
| signature | `${submail.appkey}`（环境变量配置） |

> 采用 `normal` 模式（明文 appkey），简单可靠，配合 IP 白名单 + 服务器内网调用即可保证安全。

如需更高级安全，可改用 `md5` 签名模式（需在 Submail 控制台启用）。

### 5.2 签名要求
- 短信签名：`【海峰规划】`
- 签名必须放在内容最前端
- 签名需在 Submail 控制台提前申请并审核通过

## 6. 模块 & 目录结构

```
haifeng-common/
├── service/
│   └── SmsService.java                  ← 短信发送接口
└── service/impl/
    └── SmsServiceImpl.java              ← 封装 Submail API 调用

haifeng-app/
├── controller/auth/
│   └── AppAuthController.java           ← 新增 2 个端点
├── service/auth/
│   └── AppAuthService.java             ← 新增接口方法
├── service/impl/auth/
│   └── AppAuthServiceImpl.java         ← 新增实现
└── dto/auth/
    ├── ForgotPasswordSendCodeDTO.java   ← 发送验证码请求
    └── ForgotPasswordResetDTO.java      ← 重置密码请求
```

**为什么 SmsService 放 common？**
- 虽然当前只有 app 端使用，但短信发送是基础设施能力（类似于 CaptchaService）
- 后续 admin 端如果需要短信通知，可以直接复用，无需重复实现
- 但注意：工具类级别的服务放在 common 的 service 包下，而不是 controller/dto 包

### 6.1 Redis 新增 Key 常量（RedisKeyConstant.java）

## 7. 环境配置要求

在 `application-dev.yml` / `application-prod.yml` 中添加：

```yaml
submail:
  appid: ${SUBMAIL_APPID}
  appkey: ${SUBMAIL_APPKEY}
  sign: 【海峰规划】
```

对应的 `.env` 文件中配置：
```
SUBMAIL_APPID=your_app_id
SUBMAIL_APPKEY=your_app_key
```

## 9. 错误码

| 错误码 | 说明 | 场景 |
|--------|------|------|
| 200 | 成功 | |
| 400 | 参数错误 | 验证码过期、格式错误 |
| 404 | 资源不存在 | 手机号未注册 |
| 429 | 请求过于频繁 | 60秒冷却、日限、验证码锁定 |
| 1001 | 用户不存在 | 手机号未注册（仅 reset 接口返回该码） |
| 1002 | 密码错误 | —（本功能不涉及） |
| 1400+ | 短信相关业务错误 | Submail API 返回非成功状态时 |

## 10. 边界 & 异常处理

| 场景 | 处理方式 |
|------|----------|
| Submail 服务不可用 | 捕获 HTTP 超时/异常，返回 `500`，日志记录 error |
| 同一手机号同时请求2次 | 60秒冷却 key 阻止第二次 |
| 用户在5分钟内未收到短信 | 前端可引导用户重新点击发送（需等待60秒冷却） |
| 用户更换设备操作 | 无影响，验证码绑定手机号而非设备 |
| 密码重置后旧 Token 仍有效 | 主动清除 Redis 中的 refresh token，access token 等待自然过期（2h） |
