# 竞赛证书管理 - 后端设计

> 创建日期：2026-06-06
> 范围：haifeng-app（C 端）+ haifeng-common（Mapper 扩展）
> 父模块：竞赛证书管理
> 子模块：职业技能证书列表（任务1）、科研竞赛列表（任务2、任务3）

## 1. 目标与背景

为 C 端（App）提供"竞赛证书管理"父模块下的两类数据查询能力：

- **职业技能证书列表**：基于 `t_certificate` 表的列表与详情查询。
- **科研竞赛列表**：基于 `t_competition` + `t_competition_detail` + `t_competition_major` 三表，提供列表、详情、关联专业查询，以及「专业 → 竞赛」的反向关联查询。

数据库表 V9 迁移已就绪，对应的 Entity / Mapper 已在 `haifeng-common` 中实现。本设计聚焦于 **app 模块的 Controller / Service / DTO / VO 层**，以及 `CompetitionMajorMapper` 的两个新分页联表方法。

## 2. 范围与非目标

### 范围内

- 任务1：证书表的两类查询（公开列表 + 登录详情）
- 任务2：竞赛表的三类查询（公开列表 + 登录详情 + Pro 关联专业）
- 任务3：扩展 `MajorController`，按专业 ID 分页查询关联竞赛（Pro）

### 非目标（YAGNI）

- 证书 / 竞赛的写操作（新增 / 修改 / 删除）
- 竞赛按 `comp_level` 过滤、证书按多条件组合搜索（Need 文档未要求）
- 竞赛-专业关联的写操作
- 前端展示层（本设计仅后端）

## 3. 架构概览

延续现有 `major` / `industry` 模块的「Controller → Service → ServiceImpl → Mapper」分层模式。


## 5. 数据契约

### 5.1 数据库表（参考 V9 迁移）

- `t_certificate`：`id, cert_name, category, cert_level, applicable_major, registration_time, exam_time, exam_fee, cert_intro, exam_requirements (TEXT[]), exam_arrangement, official_website, is_deleted, ...`
- `t_competition`：`id, comp_name, comp_level, registration_time, is_deleted, ...`
- `t_competition_detail`：`id, competition_id (1:1, UK), basic_info (JSONB), awards (TEXT[]), background, purposes (TEXT[]), competition_rules (JSONB), scoring_criteria (TEXT[]), notices (TEXT[]), process_guide (JSONB), awards_display (JSONB), is_deleted, ...`
- `t_competition_major`：`id, competition_id, major_id, major_name, competition_name, created_at`（无 `is_deleted`）
- `t_major`：`id, ... status=1`（按 `status=1` 过滤存在性校验）

### 5.2 索引使用

- 证书列表按 `category` 过滤 → `idx_cert_category`
- 竞赛列表 / 详情 → 主键 / 软删除索引
- 竞赛→专业：`WHERE competition_id = ?` → `idx_cm_competition`
- 专业→竞赛：`WHERE major_id = ?` → `idx_cm_major`

## 7. 关键实现要点

### 7.1 软删除过滤

- `CertificateMapper` 查询：`is_deleted = FALSE`
- `CompetitionMapper` 查询：`is_deleted = FALSE`
- `CompetitionDetailMapper`：**当前 `findByCompetitionId` 未带 `is_deleted = FALSE` 过滤**，需在 `CompetitionDetailMapper` 中新增 `findActiveByCompetitionId(Long competitionId)` 方法带 `is_deleted = FALSE` 条件，Service 层统一调用新方法（详见 6.3 与 10.1）
- `CompetitionMajorMapper`：表无 `is_deleted` 字段，不过滤
- `t_major`：`status = 1`（MajorController 新接口的 majorId 需在校验时按 status=1 过滤）

### 7.2 排序

- 证书 / 竞赛列表：`ORDER BY id ASC`（与 industry 模块一致）
- 关联查询（2.3 / 3.1）：`ORDER BY cm.id ASC`（利用现有索引顺序）

### 7.3 JSONB 反序列化

`CompetitionDetail` 实体已使用 `@TableField(typeHandler = JacksonTypeHandler.class)`，VO 字段类型与实体保持一致即可（`Map<String,Object>` / `List<Map<String,String>>` / `List<String>`）。无需在 Service 层手工 JSON 解析。

### 7.4 错误处理

- 详情接口（1.2 / 2.2）主表无记录 → `BusinessException(ResultCode.NOT_FOUND, "证书不存在" / "竞赛不存在")`
- 关联接口（2.3 / 3.1）：
  - `competitionId` 在 `t_competition` 不存在 → `BusinessException(ResultCode.NOT_FOUND, "竞赛不存在")`
  - `majorId` 在 `t_major` 不存在（status=1）→ `BusinessException(ResultCode.NOT_FOUND, "专业不存在")`
  - 关联表无数据 → 返回空 `IPage`，不抛错（与 `MajorServiceImpl.page` 风格一致）
- 列表参数 category 无匹配 → 返回空 `IPage`

### 7.5 联表 SQL（Mapper 扩展）

### 8.2 MajorServiceImpl 实现

```java
@Override
public IPage<CompetitionBriefVO> competitions(Long majorId, BasePageQueryDTO dto) {
    // 1. 校验专业存在且上架
    Major major = majorMapper.selectOne(new LambdaQueryWrapper<Major>()
            .eq(Major::getId, majorId)
            .eq(Major::getStatus, STATUS_PUBLISHED));
    if (major == null) {
        log.debug("专业不存在或已下架, majorId={}", majorId);
        throw new BusinessException(ResultCode.NOT_FOUND, "专业不存在");
    }
    // 2. 联表分页
    Page<Map<String, Object>> page = new Page<>(dto.getPage(), dto.getSize());
    IPage<Map<String, Object>> mapPage =
            competitionMajorMapper.selectCompetitionsByMajorId(page, majorId);
    return mapPage.convert(row -> CompetitionBriefVO.builder()
            .competitionId(row.get("competitionId") != null
                    ? ((Number) row.get("competitionId")).longValue() : null)
            .competitionName(row.get("competitionName") != null
                    ? String.valueOf(row.get("competitionName")) : null)
            .build());
}
```

> 注：需在 `MajorServiceImpl` 注入 `CompetitionMajorMapper`。

## 10. 风险与权衡

### 风险 1：CompetitionDetailMapper.findByCompetitionId 未过滤 is_deleted

**已确认现状**：`findByCompetitionId` 的 SQL 为 `SELECT * FROM t_competition_detail WHERE competition_id = #{competitionId}`，未带 `is_deleted = FALSE`。

**处理方式**：在 `CompetitionDetailMapper` 中新增 `findActiveByCompetitionId(Long competitionId)` 方法，SQL 加上 `AND is_deleted = FALSE`。Service 层任务2接口2 调用新方法。该方法比"先查主表再查详情"更直接，且与 spec 6.3 中明确的修改文件保持一致。

**竞赛详情接口（任务2接口2）的 Service 流程**：
1. 通过 `competitionMapper.selectOne` 校验主表存在且 `is_deleted = FALSE`，不存在则抛 `BusinessException(NOT_FOUND, "竞赛不存在")`。
2. 通过 `competitionDetailMapper.findActiveByCompetitionId(competitionId)` 查详情，可能为 null。
3. 若详情为 null：返回 `CompetitionDetailVO` 仅含 `id, competitionId`（其余字段为 null），不抛错（与 Need 文档精神"前端动态展示数据表里的数据"一致）。

### 风险 2：JSONB 字段类型动态

前端需按 `basic_info` / `competition_rules` / `process_guide` / `awards_display` 的具体结构自行解析。已在 Need 文档"注意事项"中确认"表中 JSONB 字段直接返回，由前端渲染"。

### 风险 3：竞赛-专业关联表无 `is_deleted`

竞赛或专业被逻辑删除后，关联记录仍存在。本设计不加额外过滤以保持最小变更（与 Need 中"不修改迁移"原则一致）。如果未来需过滤，需要删除或迁移关联数据。

### 权衡

- 严格遵守 Need 文档要求，未增加按 `comp_level` / `cert_level` / 关键字搜索等"可能有用"的能力。
- 复用现有 `BasePageQueryDTO`，不引入新分页参数。
